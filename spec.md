# Проект: Reciper
**Описание:** Мобильный AI-ассистент питания, генерации рецептов и пошагового приготовления блюд. Цель приложения — избавить пользователя от рутины подсчета КБЖУ, выбора блюд и инвентаризации холодильника.

## 1. Технический стек
### 1.1. Backend
*   **Язык/Фреймворк:** Python 3.11+, FastAPI.
*   **База данных:** PostgreSQL (через SQLAlchemy или SQLModel + Alembic).
*   **Кеширование/Блокировки:** Redis (использование Redlock для предотвращения спама ресурсоемкими AI-запросами от одного пользователя).
*   **Архитектура:** Clean Architecture (Domain -> Application/Services -> Infrastructure -> API/Routers).

### 1.2. Frontend (Мобильное приложение)
*   **Язык/Фреймворк:** Dart, Flutter.
*   **Стейт-менеджмент:** Riverpod.
*   **Навигация:** GoRouter (или AutoRoute).
*   **UI/Графики:** `fl_charts` (для статистики КБЖУ).
*   **Архитектура:** Feature-first Clean Architecture (`feature_name/` -> `data`, `domain`, `presentation`).

### 1.3. Инфраструктура ИИ (AI & ML)
*   **Среда:** Docker.
*   **Ограничение железа:** MVP работает на локальном ноутбуке с NVIDIA RTX 3060 (6GB VRAM).
*   **Модель:** Легкая Vision-модель (например, LLaVA 1.5 7B или Moondream2) в квантовании GGUF 4-bit.
*   **Сервер ИИ:** Ollama или llama.cpp-server, запущенный в Docker (с поддержкой CUDA). Бэкенд общается с сервером по REST API (совместимому с форматом OpenAI API).

## 2. Ключевые процессы и Интеграции

### 2.1. Генерация картинок блюд (Workaround)
Из-за нехватки VRAM генерация изображений еды через Stable Diffusion запрещена.
**Решение:** Нейросеть генерирует в JSON-ответе поле `search_query` (например, `"tomato cheese omelet"`). Backend делает запрос к **API Unsplash** по этому запросу и отдает Frontend'у URL готовой красивой фотографии блюда. В случае ошибки API используется локальная заглушка (`assets/default_food.png`).

### 2.2. Асинхронная обработка (Polling)
Так как инференс модели на 6GB VRAM занимает время, API должно быть асинхронным:
1. Мобилка шлет POST с картинкой.
2. Бэкенд возвращает `task_id` и статус `processing`.
3. Мобилка делает GET-запросы каждые 3-5 секунд (показывая анимацию загрузки).
4. Бэкенд возвращает готовый рецепт со статусом `done`.

## 3. Базовые сущности (Domain Models)

1.  **User (Пользователь):** `id`, `name`, `daily_calories_target`, `target_protein`, `target_fat`, `target_carbs`.
2.  **Recipe (Рецепт):** `id`, `title`, `description`, `image_url` (из Unsplash), `prep_time_minutes`, `calories`, `protein`, `fat`, `carbs`.
3.  **RecipeStep (Шаг рецепта):** `id`, `recipe_id`, `step_number`, `instruction`, `timer_seconds` (опционально).
4.  **DailyStat (Статистика за день):** `id`, `user_id`, `date`, `total_calories`, `total_protein`, `total_fat`, `total_carbs`.

---

## 4. Спецификация Backend API (FastAPI)

### Роутеры (API Layer):
*   `POST /api/v1/fridge/scan` — Принимает `multipart/form-data` (фотографию). Ставит задачу в фон, возвращает `task_id`.
*   `GET /api/v1/tasks/{task_id}` — Возвращает статус задачи. Если статус `done`, возвращает JSON со списком распознанных продуктов и списком сгенерированных `Recipe` (обычно 3-5 вариантов).
*   `POST /api/v1/meals/consume` — Принимает `recipe_id` съеденного блюда и добавляет его КБЖУ в `DailyStat` пользователя на текущую дату.
*   `GET /api/v1/users/{user_id}/stats` — Возвращает агрегированные данные по КБЖУ за неделю/месяц для отрисовки графиков на клиенте.

## 5. Спецификация Frontend (Flutter Screens)

1.  **Main / Profile Screen (Главный экран / Статистика)**
    *   Верхняя часть: `PieChart` (fl_charts) с текущим потреблением БЖУ за сегодня.
    *   `LineChart` (fl_charts) с потреблением калорий за последние 7 дней относительно линии цели (Target).
    *   Кнопка (FAB) с иконкой камеры — "Сканировать холодильник".
2.  **Scanner Screen (Экран сканера)**
    *   Интерфейс камеры (или выбор из галереи).
    *   После отправки фото: Экран с красивой Lottie/Rive анимацией ("Нейросеть изучает холодильник...") — в этот момент работает Polling статуса `task_id`.
3.  **Recipe Selection Screen (Выбор блюда)**
    *   Отображение списка карточек рецептов, которые вернул бэкенд.
    *   Карточка: Изображение (network image из Unsplash), Название, КБЖУ, Время готовки.
    *   Большая зеленая кнопка "Готовить" на каждой карточке.
4.  **Cooking Mode Screen (Режим готовки)**
    *   Реализован через `PageView` (постраничная навигация по шагам).
    *   Крупный шрифт, контрастные цвета (чтобы видеть издалека).
    *   Если в шаге есть время (`timer_seconds`), отображается виджет кругового таймера с кнопкой Play/Pause.
    *   На последнем шаге кнопка "Приятного аппетита!" (отправляет POST-запрос на обновление статистики).

## 6. Требования к написанию кода для ИИ

1.  **Strict Clean Architecture:** Для бэкенда создавай папки `domain`, `infrastructure`, `repositories`, `services`, `api`. Запрещено писать SQL-запросы или вызовы HTTP/AI напрямую в роутерах (FastAPI endpoints). Используй Dependency Injection (например, `Depends` в FastAPI).
2.  **Pydantic & Types:** Строгая типизация во всем Python коде.
3.  **Flutter State:** Для управления состоянием используй `StateNotifierProvider` или новые аннотации `@riverpod` из пакета `riverpod_generator`. Отделяй бизнес-логику от UI.
4.  **Обработка ошибок:** Предусмотреть обработку ошибок, если нейросеть вернула невалидный JSON, или API Unsplash не нашло картинку (fallbacks). 
5.  **Документация:** Оставлять короткие докстринги к основным интерфейсам (abstract classes).
